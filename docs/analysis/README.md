## 🎯 主要な発見事項

### 1. HTTPS→HTTP→HTTPSの二重リダイレクト

**現状**:
```bash
$ curl -I https://nxp.nyle.co.jp/marketing
HTTP/2 301 → http://nxp.nyle.co.jp/marketing/  # nginx
HTTP/1.1 301 → https://nxp.nyle.co.jp/marketing/ # ALB
HTTP/2 200                                       # nginx
```

**原因**: nginxがALBのHTTPS終端を認識せず、HTTPスキームでリダイレクト

**影響**: 150-250msのレスポンス遅延、SEOクロール効率低下

### 2. 設定の分散（6ファイル、5箇所）

| 設定箇所 | 状態 | リダイレクト例 |
|---------|------|---------------|
| ALB Rules | ✅ 使用中 | `/` → `/marketing/` (301) |
| nginx `10-nxp-prod.conf` | ✅ 使用中 | `/marketing` → `/marketing/` (301) |
| nginx `nxp.conf` | ❓ 未使用？ | `/` → `/marketing/` (302) |
| CloudFront Function | ❓ 未使用？ | `/` → `/marketing/` (302) |
| HTML meta refresh | ❓ バックアップ？ | JavaScript redirect |

### 3. 301 vs 302の混在

- ALB: 301（正しい）
- nginx現行: 301（正しい）
- nginx旧: 302（SEO的に不適切）
- CloudFront: 302（SEO的に不適切）

## 🛠️ 推奨ソリューション（概要）

### アプローチ: 責任範囲の明確化

```
┌─────────────────────────────────────────┐
│ ALB                                      │
│ - プロトコル正規化（HTTP→HTTPS）         │
│ - URL構造正規化（パスリダイレクト）       │
│ - Trailing slash正規化                   │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ nginx                                    │
│ - SPAルーティング（try_files）            │
│ - キャッシュ制御                          │
│ - セキュリティヘッダー                    │
└─────────────────────────────────────────┘
```

### 実装フェーズ

| Phase | 内容 | 所要時間 | 効果 |
|-------|------|---------|------|
| **Phase 1** | 二重リダイレクト解消 | 1-2時間 | パフォーマンス改善（150-250ms） |
| **Phase 2** | 設定整理、ドキュメント作成 | 2-3時間 | 保守性向上 |
| **Phase 3** | 最適化、IaC導入 | 1日 | 長期的な運用効率化 |

## ⚠️ リスクと軽減策

| リスク | 発生確率 | 影響度 | 軽減策 |
|-------|---------|--------|--------|
| 二重リダイレクト再発 | 低 | 中 | 自動テスト導入、監視アラート |
| ALB設定ミス | 中 | 高 | ステージング検証、IaC化 |
| SEO順位一時低下 | 低 | 中 | 301使用、GSC監視 |
| ユーザーアクセス不可 | 低 | 高 | 段階的リリース、即座にロールバック可能 |
